Response #1: {Part 1: Ambiguities, missing clarifications, risks, and edge cases

Your spec is already pretty good, but there are several places where an implementer could make different choices and still claim they followed it.

1. Ambiguities in status/result values

The spec says the system returns “the affected user’s resulting status,” but it does not define the full set of allowed status values.

Unclear points:

What exact statuses exist?

registered

waitlisted

not_registered

not_found

cancelled

duplicate

promoted

Are not_registered and not_found the same thing or different things?

After a cancel, should the affected user’s status be reported as not_registered or cancelled?

For a duplicate register attempt, should the result be:

unchanged registered,

unchanged waitlisted,

or an error-like status such as already_registered?

This needs a fixed vocabulary.

2. Ambiguity about query input and output semantics

The input list says user_id is always provided, but query could be interpreted in two ways:

query a specific user’s status, or

query full system state

The functional requirements suggest user-specific query, but the outputs always include the full registered and waitlisted lists. That is okay, but it should be stated explicitly:

query(user_id) returns that user’s status plus full state.

Also unclear:

Is query valid for users not in the system?

The assumptions say return not_registered or not_found, which again is ambiguous.

3. Ambiguity about ordering of the registered list

The spec requires the waitlist to be FIFO, but it never clearly defines what order the registered list should preserve.

Possible interpretations:

registration order among currently registered users

current occupancy order after promotions

sorted by user_id

unspecified but deterministic

Example:

capacity 2, registered [A, B], waitlist [C]

A cancels, C is promoted

What should the registered list become?

[B, C] if preserving current occupancy order

[C, B] if promoted user fills first “vacancy”

some other deterministic order

Your AC3 implies D remains first in waitlist, but does not explicitly define registered ordering after promotion.

This must be specified.

4. Ambiguity about duplicate registration attempts from waitlisted users

“The system shall prevent duplicate registrations” is not precise enough.

Questions:

If a user is already waitlisted and calls register again, what happens?

Is it treated as:

no-op with status waitlisted,

rejection with already_waitlisted,

or does it refresh their place in line?
The FIFO requirement suggests it must not refresh position, but that should be explicit.

5. Ambiguity about cancellation semantics

The system shall allow registered and waitlisted users to cancel, but for cancellation of a user not present:

Is cancel a no-op?

Does it return not_registered?

Does it return not_found?

Is this considered an error?

It says “return a consistent not_registered or not_found result,” but that wording leaves two valid implementations.

That should be narrowed to exactly one required outcome.

6. Underspecified behavior for capacity = 0

You say the system may place the user on the waitlist according to the stated assumption.

“May” is too weak for acceptance criteria. It allows multiple behaviors:

place on waitlist

reject entirely

some other interpretation

But your earlier assumption says the system shall still function consistently and allow a waitlist. That strongly suggests registration attempts should go to waitlist when capacity is 0.

So AC6 is weaker than the rest of the spec and should be strengthened from “may” to “shall”.

7. Ambiguity about promotion timing

“The system shall automatically promote the earliest waitlisted user when a registered user cancels.”

This mostly implies immediate promotion, but it should explicitly say:

promotion occurs within the same cancellation operation, before returning output.

Otherwise one implementation could:

remove the registered user first,

leave a temporary vacancy,

promote later in a separate step.

The invariant says state remains consistent after every operation, but immediate promotion should be explicit.

8. Ambiguity about system initialization / lifecycle

The spec says single event with fixed capacity, but it does not clearly define:

whether capacity is supplied once at system initialization or on every operation

whether capacity can ever change after initialization

what the initial state is besides “no users”

This matters because “input” includes capacity, which sounds like per-operation input, while the functional requirements say event capacity is fixed.

Potential contradiction:

If capacity is an input every time, a caller could submit different values on later operations.

If capacity is fixed, it should be initialization-time configuration, not normal operation input.

9. Ambiguity about what counts as a valid user_id

You say non-empty string and exact match, but still unclear:

Are leading/trailing spaces significant?

Is "A" different from "a"? Exact match implies yes, but that should be stated as case-sensitive if intended.

Are whitespace-only strings invalid?

Are special characters allowed?

At minimum, define:

non-empty means length > 0 after no normalization, or after trimming?

comparison is case-sensitive exact string equality.

10. Ambiguity about deterministic behavior

“The system shall maintain deterministic behavior.”

This is good, but determinism depends on more than FIFO. It needs all tie/order rules defined:

registered list ordering

duplicate attempt handling

cancel-not-found handling

query result terminology

Right now determinism is claimed, but a few outputs are still open-ended.

11. Missing explicit no-op rules

Several operations should probably be defined as no-op state changes:

duplicate register by registered user

duplicate register by waitlisted user

cancel by absent user

query by absent user

The spec hints at these, but does not clearly say whether system state must remain unchanged in each case.

That should be made explicit.

12. Missing explicit rule for one operation at a time

You have “operations are processed one at a time in the order received,” which is good, but it would help to say:

there is no concurrent execution within the scope of this specification.

That makes the single-threaded assumption explicit.

13. Possible contradiction: “not_registered” vs “not_found”

These are used interchangeably in assumptions, but they do not necessarily mean the same thing:

not_found could mean “user was never seen”

not_registered could mean “user is currently not in system”

Since the system does not appear to maintain user history, not_registered is the cleaner choice.

Using both creates inconsistency.

14. Possible contradiction: “prevent duplicate registrations” vs “allow re-registration after cancellation”

This is not a real contradiction, but it needs sharper wording:

duplicates are forbidden only while the user is currently present in the system

once fully removed, they may register again

That distinction should be stated directly.

15. Acceptance criteria are incomplete for several key cases

The ACs cover many basics, but not all important branches. Missing or weak acceptance tests include:

duplicate register while already waitlisted

cancel registered user when waitlist empty

cancel user not in system

query user not in system

re-register after cancelling from waitlist

exact registered-list ordering after promotion

duplicate register does not change waitlist position

capacity fixed at initialization and not modifiable later

16. Risk: full-state output may expose internal data not always desired

This is not necessarily wrong, but every operation returns full lists. That is okay for a lab spec, but note that it is a design choice and should be stated as intentional.

17. Edge cases that should be clarified before implementation

These are the biggest ones to pin down:

What exact status values are allowed?

Is cancel on an absent user a no-op with not_registered?

Is query on an absent user not_registered?

What happens when an already waitlisted user registers again?

What exact order should the registered list maintain?

When promotion occurs, is it immediate during the same operation?

Is capacity passed once at initialization only?

Is user_id comparison case-sensitive?

Are whitespace-only IDs invalid?

For capacity = 0, does every successful register attempt go to waitlist?

Revised Specification: Event Registration with Waitlist
1. System objective

Design a deterministic system for a single event with a fixed capacity. The system shall:

register users while capacity is available,

place additional users on a waitlist when capacity is full,

automatically promote waitlisted users in FIFO order when registered users cancel.

The system processes one operation at a time in the order received.

2. System model

The system manages exactly two ordered collections:

Registered list: users currently registered for the event

Waitlisted list: users waiting for a spot

A user may appear in at most one of these lists at any time.

The event capacity is fixed for the lifetime of the system instance.

3. Initialization
Input at initialization

capacity: integer, capacity >= 0

Initial state

registered list = empty

waitlisted list = empty

Capacity is configured once at initialization and shall not change afterward.

4. Operation inputs

Each operation consists of:

operation: one of {register, cancel, query}

user_id: a string identifying exactly one user

User ID validity rules

user_id must be a non-empty string

comparison is by exact string match

matching is case-sensitive

no normalization is performed unless explicitly stated elsewhere

For this specification, "A" and "a" are different users.

5. Output after every operation

After each operation, the system shall return:

the affected user’s resulting status,

the ordered registered list,

the ordered waitlisted list.

Allowed status values

The resulting status for the affected user shall be exactly one of:

registered: user is currently in the registered list

waitlisted: user is currently in the waitlisted list

not_registered: user is currently in neither list

No other status terms shall be used.

6. Ordering rules
Registered list ordering

The registered list shall preserve the order in which users most recently entered the registered state.

A user registered directly into the event is appended to the end of the registered list.

A user promoted from the waitlist is appended to the end of the registered list.

Removing a registered user preserves the relative order of the remaining registered users.

Waitlist ordering

The waitlist shall preserve FIFO order.

A user added to the waitlist is appended to the end of the waitlist.

Promotion removes the earliest waitlisted user.

Cancellation by a waitlisted user removes that user and preserves the relative order of all remaining waitlisted users.

7. Operation semantics
7.1 Register operation

When register(user_id) is processed:

Case R1: user already registered

If the user is already in the registered list:

the system shall not create a duplicate,

the system state shall remain unchanged,

the returned status shall be registered.

Case R2: user already waitlisted

If the user is already in the waitlisted list:

the system shall not create a duplicate,

the user shall retain the same waitlist position,

the system state shall remain unchanged,

the returned status shall be waitlisted.

Case R3: user not currently in system and space is available

If the user is not in the system and the number of registered users is less than capacity:

the user shall be appended to the registered list,

the returned status shall be registered.

Case R4: user not currently in system and event is full

If the user is not in the system and the number of registered users equals capacity:

the user shall be appended to the end of the waitlist,

the returned status shall be waitlisted.

This rule also applies when capacity is 0.

7.2 Cancel operation

When cancel(user_id) is processed:

Case C1: user is registered

If the user is in the registered list:

the user shall be removed from the registered list,

if the waitlist is non-empty, the earliest waitlisted user shall be promoted immediately within the same operation,

the promoted user shall be removed from the front of the waitlist and appended to the end of the registered list,

the cancelling user’s returned status shall be not_registered.

Case C2: user is waitlisted

If the user is in the waitlisted list:

the user shall be removed from the waitlisted list,

no promotion shall occur,

the cancelling user’s returned status shall be not_registered.

Case C3: user is not in system

If the user is in neither list:

the system state shall remain unchanged,

the returned status shall be not_registered.

7.3 Query operation

When query(user_id) is processed:

Case Q1: user is registered

return status registered

Case Q2: user is waitlisted

return status waitlisted

Case Q3: user is not in system

return status not_registered

A query operation shall not modify system state.

8. Functional requirements

The system shall have a fixed event capacity configured at initialization.

The system shall register a user when capacity is available and the user is not already in the system.

The system shall place a user on the waitlist when capacity is full and the user is not already in the system.

The system shall maintain the waitlist in FIFO order.

The system shall automatically promote the earliest waitlisted user when a registered user cancels and the waitlist is non-empty.

The system shall prevent duplicate users from appearing anywhere in the system.

The system shall allow a registered user to cancel.

The system shall allow a waitlisted user to cancel before promotion.

The system shall allow any user to query current status.

The system shall return the updated state after every operation.

The system shall allow re-registration after a prior cancellation.

The system shall behave consistently when capacity = 0.

The system shall behave deterministically for any given initial state and operation sequence.

9. Constraints

C1. Event capacity shall be a fixed integer greater than or equal to 0.
C2. The number of registered users shall never exceed capacity.
C3. The waitlist shall preserve FIFO order.
C4. No duplicate user shall exist in the system.
C5. A user shall not exist in more than one state at the same time.
C6. When a registered user cancels and the waitlist is non-empty, the earliest waitlisted user shall be promoted immediately in the same operation.
C7. After each operation, the system shall return the ordered registered and waitlisted lists.
C8. The system shall support only the operations register, cancel, and query.
C9. A duplicate register attempt shall not change system state or queue position.
C10. A query operation shall not change system state.
C11. A cancel operation for a user not in the system shall not change system state.

10. Invariants

After every operation:

the number of registered users is less than or equal to capacity,

no user appears more than once in the system,

no user appears in both registered and waitlisted lists,

registered and waitlisted lists remain internally ordered as specified,

the system state is consistent and deterministic.

11. Negative requirements

The system shall not register more users than the configured capacity.

The system shall not allow the same user to appear more than once.

The system shall not allow the same user to exist in both registered and waitlisted lists.

The system shall not violate FIFO ordering when promoting from the waitlist.

The system shall not change a waitlisted user’s position because of a duplicate register attempt.

The system shall not modify state during a query.

The system shall not modify state when cancelling a user not in the system.

12. Assumptions

Operations are processed sequentially, one at a time, in arrival order.

The system manages one event only.

User identity is determined by exact, case-sensitive user_id match.

The system does not maintain special priority for previously cancelled users.

A user removed by cancellation is fully removed from the system and may later register again as a new incoming user.

When capacity is 0, all successful new register attempts shall be placed on the waitlist.

13. Tie-breaking rule

Operations are processed one at a time in the order received.

If promotion is required, the earliest user currently in the waitlist shall be promoted first.

No other tie-breaking rule is needed because waitlist order is total and deterministic.

14. Revised acceptance criteria

Design a deterministic system for a single event with a fixed capacity. The system shall:

register users while capacity is available,

place additional users on a waitlist when capacity is full,

automatically promote waitlisted users in FIFO order when a registered user cancels.

The system processes operations one at a time in the order received.

1.2 Event model

The system manages one event and maintains:

an ordered registered list,

an ordered waitlisted list.

Capacity is a fixed property of the event/system instance and does not change during normal operation.

1.3 Inputs
Event/system property

capacity: integer, where capacity >= 0

Operation input

Each operation consists of:

operation: one of {register, cancel, query}

user_id: non-empty string identifying one user

the affected user’s resulting status,

the ordered registered list,

the ordered waitlisted list.

Allowed user status values

The resulting status shall be exactly one of:

registered

waitlisted

not_registered

No other status values shall be used.

1.5 Assumptions

User identity is determined by exact user_id string match.

user_id comparison is case-sensitive.

Whitespace-only user_id values are invalid.

Operations are processed sequentially, one at a time, in arrival order.

A user who cancelled earlier may register again later and is treated like any new incoming user, with no special priority.

1.6 Ordering rules
Registered list ordering

The registered list preserves the order in which users currently entered the registered state.

Direct registrations append to the end of the registered list.

Promotions from the waitlist also append to the end of the registered list.

Removing a registered user preserves the relative order of the remaining registered users.

Waitlisted list ordering

The waitlist preserves FIFO order.

A new waitlisted user is appended to the end of the waitlisted list.

If a user is promoted, that user is removed from the front of the waitlisted list.

If a waitlisted user cancels, that user is removed and the relative order of the remaining waitlisted users is preserved.

When register(user_id) is processed:

If the user is already registered:

the state shall remain unchanged,

no duplicate entry shall be created,

the returned status shall be registered.

If the user is already waitlisted:

the state shall remain unchanged,

no duplicate entry shall be created,

the returned status shall be waitlisted.

the user shall be appended to the end of the registered list,

the returned status shall be registered.

If the user is not in the system and the event is full:

the user shall be appended to the end of the waitlisted list,

the returned status shall be waitlisted.

If capacity = 0, a new register attempt by a user not already in the system shall place the user on the waitlist.

When cancel(user_id) is processed:

If the user is in the registered list:

the user shall be removed from the registered list,

the user shall be removed from the waitlisted list,

no promotion shall occur,

the cancelling user’s returned status shall be not_registered.

If the user is not in the system:

the state shall remain unchanged,

the returned status shall be not_registered.

When query(user_id) is processed:

If the user is registered:

return status registered.

If the user is waitlisted:

return status waitlisted.

If the user is not in the system:

return status not_registered.

A query operation shall not modify system state.

A query(user_id) operation shall return that user’s status together with the full ordered registered list and full ordered waitlisted list.

C1. Event capacity shall be a fixed integer greater than or equal to 0.
C2. The number of registered users shall never exceed capacity.
C3. The waitlist shall preserve FIFO order.
C4. There shall be no duplicate users in the system.
C5. A user shall not exist in more than one state at the same time.
C6. When a registered user cancels and the waitlist is non-empty, the earliest waitlisted user shall be promoted immediately within the same cancel operation.
C7. The system shall return ordered registered and waitlisted lists after each operation.

After every operation:

the number of registered users never exceeds capacity,

no user appears more than once in the system,

no user exists simultaneously in both registered and waitlisted states,

the system state remains consistent,

The system shall not allow the same user to appear twice in the system.

The system shall not place a user in both registered and waitlisted lists simultaneously.

The system shall not violate FIFO ordering when promoting from the waitlist.

The system shall not change state on a duplicate register attempt.

The system shall not change state when cancelling a user not in the system.

The system shall not change state during a query.

1.11 Tie-breaking and determinism

Operations are processed one at a time in the order received.

If multiple users are eligible for promotion, the user who entered the waitlist first must be promoted first.

For a given initial state and the same sequence of operations, the system shall produce the same outputs and final state.

class EventRegistrationSystem:
    """
    Deterministic event registration system for a single event with a fixed capacity.

    Status values are exactly:
    - registered
    - waitlisted
    - not_registered
    """

    def __init__(self, capacity: int) -> None:
        if not isinstance(capacity, int):
            raise TypeError("capacity must be an integer")
        if capacity < 0:
            raise ValueError("capacity must be >= 0")

        self._capacity = capacity
        self._registered: List[str] = []
        self._waitlisted: List[str] = []

    @property
    def capacity(self) -> int:
        return self._capacity

    def register(self, user_id: str) -> OperationResult:
        self._validate_user_id(user_id)

        current_status = self._status_of(user_id)

        # C4, C5, C9: duplicate register attempts are no-ops
        if current_status == REGISTERED:
            return self._result(REGISTERED)

        if current_status == WAITLISTED:
            return self._result(WAITLISTED)

            # Direct registrations append to registered list
            self._registered.append(user_id)
            return self._result(REGISTERED)

        return self._result(WAITLISTED)

        if user_id in self._registered:
            # Remove registered user
            self._registered.remove(user_id)

            # C6: immediate promotion in the same cancel operation
            if self._waitlisted:

                self._registered.append(promoted)

            return self._result(NOT_REGISTERED)

        if user_id in self._waitlisted:
            self._waitlisted.remove(user_id)
            return self._result(NOT_REGISTERED)

        # C11: absent-user cancel is a no-op
        return self._result(NOT_REGISTERED)

        self._validate_user_id(user_id)

        # C10: query must not modify state
        return self._result(self._status_of(user_id))

        if user_id in self._registered:
            return REGISTERED
        if user_id in self._waitlisted:
            return WAITLISTED
        return NOT_REGISTERED

        return OperationResult(
            status=status,

            waitlisted=list(self._waitlisted),

class EventRegistration:
    """
    Students must implement this class per the lab handout.
    Deterministic ordering is required (e.g., FIFO waitlist, predictable registration order).
    """

    def __init__(self, capacity: int) -> None:
        """
        Args:
            capacity: maximum number of registered users (>= 0)
        """
        if not isinstance(capacity, int):
            raise TypeError("capacity must be an integer")
        if capacity < 0:
            raise ValueError("capacity must be >= 0")

        self._capacity = capacity
        self._registered: List[str] = []
        self._waitlist: List[str] = []

    def register(self, user_id: str) -> UserStatus:
        """
        Register a user:
          - if capacity available -> registered
          - else -> waitlisted (FIFO)

        Raises:
            DuplicateRequest if user already exists (registered or waitlisted)
        """
        if user_id in self._registered or user_id in self._waitlist:
            raise DuplicateRequest(f"{user_id} is already in the system")

        if len(self._registered) < self._capacity:
            self._registered.append(user_id)
            return UserStatus("registered")

        self._waitlist.append(user_id)
        return UserStatus("waitlisted", len(self._waitlist))

    def cancel(self, user_id: str) -> None:
        """
        Cancel a user:
          - if registered -> remove and promote earliest waitlisted user (if any)
          - if waitlisted -> remove from waitlist
          - behavior when user not found depends on handout (raise NotFound or ignore)

        Raises:
            NotFound (if required by handout)
        """
        if user_id in self._registered:
            self._registered.remove(user_id)

            if self._waitlist and len(self._registered) < self._capacity:
                promoted = self._waitlist.pop(0)
                self._registered.append(promoted)
            return

        if user_id in self._waitlist:
            self._waitlist.remove(user_id)
            return

        raise NotFound(f"{user_id} not found")

    def status(self, user_id: str) -> UserStatus:
        """
        Return status of a user:
          - registered
          - waitlisted with position (1-based)
          - none

        if user_id in self._registered:
            return UserStatus("registered")

        if user_id in self._waitlist:
            return UserStatus("waitlisted", self._waitlist.index(user_id) + 1)

        return UserStatus("none")

        """
        (Optional helper for debugging/tests)

# Validates FIFO promotion order under multiple cancellations and deterministic behavior.
def test_multiple_cancellations_promote_in_fifo_order():

    er.register("u1")
